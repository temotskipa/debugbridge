# DebugBridge Fabric 26.2-snapshot-3 Design

## Goal

Add a new `fabric-26.2-snapshot-3` module that gives Minecraft `26.2-snapshot-3` full runtime parity with the existing `fabric-1.21.11` DebugBridge implementation so `mcdev-mcp` can use the same DebugBridge-backed capabilities without per-version branching.

## Current Context

The repository already separates shared bridge logic from version-specific Fabric integration:

- `mod/core/` contains the shared WebSocket server, Lua runtime, mapping logic, protocol types, screenshot helpers, and provider interfaces.
- `mod/fabric-1.19/` and `mod/fabric-1.21.11/` each implement the version-specific Fabric layer for a supported Minecraft target.
- `web-ui/` consumes the same request and response shapes regardless of Minecraft version.

There was also an older `fabric-26.1` module in repository history, but it was removed before the modern runtime-provider surface was added. It is useful only as evidence that a 26.x module belongs beside the existing versioned modules, not as the implementation baseline.

## Architecture

The new target should be implemented as a sibling module at `mod/fabric-26.2-snapshot-3/`, modeled directly on `mod/fabric-1.21.11/`.

The key architectural rule is that `mod/core/` remains the stable contract layer. The new module is an adapter that satisfies the same provider interfaces and request surface already used by `fabric-1.21.11`. The first pass should avoid broad shared-code refactors and should keep snapshot-specific churn isolated inside the new module unless a small compatibility change in `core/` is forced by the port.

## Module Layout

The new module should include the same version-specific responsibilities as `fabric-1.21.11`:

- `DebugBridgeMod.java`
- `DeveloperWarningScreen.java`
- `FabricMojangResolver.java`
- `Minecraft262Snapshot3ScreenshotProvider.java`
- `Minecraft262Snapshot3NearbyEntitiesProvider.java`
- `Minecraft262Snapshot3LookedAtEntityProvider.java`
- `Minecraft262Snapshot3ItemTextureProvider.java`
- `mixin/MinecraftClientMixin.java`
- `mixin/EntityGlowMixin.java`
- `src/main/resources/fabric.mod.json`
- `src/main/resources/debugbridge.mixins.json`
- `src/main/resources/assets/debugbridge/icon.png`

This keeps the version-specific surface area explicit and aligned with the current 1.21.11 implementation, which makes parity and regressions easier to reason about.

## Dependency Strategy

The module should use snapshot-capable tooling instead of inheriting 1.21.11-era assumptions:

- Minecraft: `26.2-snapshot-3`
- Fabric Loader: current compatible 26.2-capable line, presently `0.19.2`
- Fabric API: current `26.2` line, presently `0.146.0+26.2`
- Fabric Loom: a current alpha that supports 26.2 snapshots, presently `1.17.0-alpha.6`
- Java: target the snapshot-required runtime and validate the exact build requirement during implementation

The implementation should pin concrete versions in the module build file and update shared Gradle wiring only where necessary to support the new module.

## Parity Definition

Full parity with `fabric-1.21.11` means `mcdev-mcp` can treat `26.2-snapshot-3` as a first-class runtime target with no version-specific branching in its DebugBridge usage.

The module must support:

- developer warning and acceptance gating
- localhost WebSocket bridge startup
- Lua execution using Mojang names
- snapshot capture
- screenshot capture
- nearby entity listing
- entity details lookup
- looked-at entity selection
- glow outline for selected entities
- item texture extraction
- entity item texture extraction

Parity is defined by runtime behavior and response shape, not by identical code structure.

## Implementation Approach

The implementation should follow a copy-and-port strategy:

1. Copy `fabric-1.21.11` into a new `fabric-26.2-snapshot-3` module.
2. Wire the new module into Gradle settings, build logic, and release automation.
3. Port version-specific code in vertical slices until the runtime surface matches 1.21.11.

This is preferred over reviving the removed `fabric-26.1` module because that module predates the current provider surface and would require rebuilding most of the modern functionality anyway.

It is also preferred over refactoring shared abstractions first because the immediate need is a working 26.2 snapshot target, not a generalized cross-version architecture cleanup.

## Rollout Slices

### Slice 1: Module Boot

Add the new module to:

- `mod/settings.gradle.kts`
- root build/release workflows
- module-local `build.gradle.kts`
- module resources and mixin config

Port `DebugBridgeMod` and `DeveloperWarningScreen` first, preserving:

- developer warning behavior
- port selection behavior
- bridge startup flow
- provider registration into `BridgeServer`

### Slice 2: Hook Parity

Port the same two client mixins used by 1.21.11:

- inject at `Minecraft.tick()` tail to run `DebugBridgeMod.onClientTick(...)`
- inject into `Entity.isCurrentlyGlowing()` to force glow for selected entity IDs

Port the looked-at entity provider using the snapshot version of `ProjectileUtil.getEntityHitResult(...)`.

### Slice 3: Runtime Data Providers

Port providers in this order:

1. screenshot provider
2. nearby entities provider
3. looked-at entity provider if not already complete
4. item texture provider

The screenshot and entity providers validate render-thread access and general runtime integration without taking on the full risk of offscreen GUI rendering immediately.

The item texture provider should be ported last because it is the most render-pipeline-sensitive part of the implementation.

### Slice 4: Build And Release Parity

Update build and release automation so the new module is built and published alongside the existing supported versions without regressing `1.19` or `1.21.11`.

## Known Port Risks

The major risks are concentrated in snapshot-specific API drift inside the versioned module, not in the overall project structure.

Expected low-to-medium risk areas:

- warning screen integration
- tick-tail mixin
- entity glow mixin
- nearby-entity iteration
- looked-at entity raycast
- snapshot JSON assembly

Expected higher-risk areas:

- game-thread dispatch mechanics
- screenshot capture integration
- offscreen item rendering and GPU readback
- any reflection-based access to `GuiRenderer` internals

The implementation should assume that render-backed providers may need real API adaptation instead of a blind copy from 1.21.11.

## Verification Strategy

Verification should happen slice by slice rather than waiting for one final integration check.

Required gates:

- build gate: `./gradlew :fabric-26.2-snapshot-3:build`
- boot gate: launch Minecraft with the new module and confirm warning-screen and server-start behavior
- runtime gate: exercise each WebSocket request type directly and compare output shape to 1.21.11
- `mcdev-mcp` gate: connect `mcdev-mcp` to a live `26.2-snapshot-3` client and verify runtime-dependent tools end to end

The module is not complete until all runtime-dependent `mcdev-mcp` capabilities that currently rely on DebugBridge work against `26.2-snapshot-3`.

## Failure Handling

If a provider is blocked by snapshot API churn, isolate that provider and continue bringing the rest of the module to parity rather than destabilizing shared code.

Small targeted adjustments to `mod/core/` are acceptable if the port requires them, but broad refactors are out of scope for the initial parity push.

If item textures or screenshot capture fail while the rest of the bridge works, the module should still be treated as incomplete because the stated goal is full runtime parity for `mcdev-mcp`, not partial availability.

## Out Of Scope

The first parity pass does not include:

- redesigning the provider interfaces in `mod/core/`
- web UI redesign or protocol changes
- deduplicating all shared code between version modules
- replacing existing supported modules

Those can be revisited after `26.2-snapshot-3` is working as a first-class target.
